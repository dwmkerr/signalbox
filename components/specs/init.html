signalbox init proposed

The guided setup: how it decides between writing your config and printing a snippet, and how it says which.
Proposed. This redesigns the interactive setup and adds config editing behind consent. Today init only prints blocks for user-owned files and shows a misleading "Applying... Done". The mockups below are the spec for the new states and copy.

The problem

Toggling a print-only integration reads as if it configured itself:

signalbox init (today)Applying... ○ Codex - merge the JSON block below into ~/.codex/hooks.json { ...block the user must paste... } ✔ Done. Open a session and press ⌃⌥J.

"Done" after nothing was written, and Codex still shows ○ (not set up). The output contradicts itself.

The model: classify by ownership

What init does depends on who owns the file. Three cases, drawn from how conda, rustup, gcloud, Homebrew, pre-commit and Claude Code behave:

CaseExamplesWhat init does
1. signalbox owns itthe OpenCode / pi plugin symlinks, signalbox's own configWrite it directly, no prompt, show ✔ Configured. Like git config / claude mcp add.
2. You own it, we understand it, nothing else manages it~/.tmux.conf; a plain ~/.claude/settings.json or hooks.jsonOffer to edit behind a named-file opt-in: preview, backup, managed block, idempotent, reversible. Decline → print, and the row stays not done.
3. Another tool manages itCodex's hooks.json under koi (trust hashes)Print by default. A blind rewrite breaks the trust hash and Codex skips the hook. Edit only if signalbox also refreshes the trust ledger.

The evidence, in one line: consent is the trigger to edit; a managed marker, idempotent replace-in-place, a backup, and a working --reverse are the license. Tools that asked-and-wrote without those (nvm, atuin, husky) are the ones users complain about; gh got consent to edit ~/.gitconfig but never reverts it on logout - an orphaned edit we must not copy.

The setup screen

Every row carries an honest state. Nothing is ever a check until it is actually written or verified.

signalbox initsignalbox ● Menu bar app running (hub + jumplist) Integrations ○ Claude Code not set up ● Cursor hooks active ❯ ◉ Codex set up › ● VS Code jump-back (automatic) ○ OpenCode not set up ○ pi not set up ● tmux active ↑↓ move · space toggle · ↵ apply · d details · q quit

States: ● configured · ◉ selected to apply · ○ not set up. A row never shows a check until its file is written.

Applying, with a per-component decision

For a case-2 or case-3 file, applying pauses on a choice that names the exact file (the rustup / gcloud / fzf pattern), never a silent write:

signalbox init · applying CodexCodex hooks live in ~/.codex/hooks.json. This file is managed by koi (it records Codex trust hashes), so editing it can break trust. Recommended: print the snippet and add it yourself. [P] Print the snippet to paste (recommended here) [E] Edit the file for me (backup first, then refresh trust) [S] Skip ›

Case 3 (koi-managed) defaults to Print. A plain case-2 file (tmux.conf) leads with [E] Edit instead, since it is safe to write.

After an edit (case 2, or case 3 when trust can be refreshed)

signalbox init · wrote the file✔ Wrote ~/.tmux.conf backup: ~/.config/signalbox/backups/tmux.conf.2026-07-21T08-40-11 Undo anytime: signalbox init --reverse

After a print (declined edit, or koi-managed)

signalbox init · printed a snippet⚠ Codex needs one manual step. Add this to ~/.codex/hooks.json: { "hooks": { "Stop": [{ "hooks": [{ "type": "command", "command": "signalbox hook codex" }] }], ... SessionStart, UserPromptSubmit, PermissionRequest, SessionEnd ... } } Then run /hooks in Codex to trust it.

The summary reconciles - it never claims blanket success

signalbox init · done2 configured · 1 needs a manual step · 1 skipped Codex - snippet printed above, add it to ~/.codex/hooks.json Run signalbox status to see what is left.

A component that was only printed stays ◉/○ and counts under "needs a manual step", never under configured. This is the whole fix for the misleading "Done".

How an edit is written

Text files: a managed sentinel block

Comment-capable files (.tmux.conf, shell rc) get a fenced block, matched and replaced whole on every run so re-running updates in place (the conda / pnpm idiom):

~/.tmux.conf# ... your existing config, untouched ... # >>> signalbox managed >>> # !! Managed by `signalbox init`. Edit: signalbox init Remove: signalbox init --reverse !! set -g status-right '#(command -v signalbox >/dev/null && signalbox tmux status) %Y-%m-%d %H:%M' bind-key j display-popup -E -w 80% -h 15 "signalbox pick" # <<< signalbox managed <<<

Where the body is more than a line, inject a single stable pointer into a signalbox-owned file (the rustup source ~/.cargo/env trick), so the footprint in your file stays tiny and stable across versions.

JSON files: keyed entries + a sidecar manifest

JSON holds no comments, so signalbox never fakes a marker key inside another tool's schema. Instead it (a) names its own entries recognisably - a hook whose command is signalbox hook codex - and (b) records authoritative ownership in a sidecar it owns:

~/.config/signalbox/managed.json[ { "file": "~/.codex/hooks.json", "managedKeys": ["hooks.Stop[sb]", "hooks.SessionStart[sb]", "..."], "backup": "~/.config/signalbox/backups/hooks.json.2026-07-21T08-40-11", "externalOwner": "koi" } ]

The manifest is what makes a JSON edit both idempotent (compare by managedKeys, replace in place) and cleanly removable (delete exactly those keys, restore the backup if nothing of ours remains). It is also where the koi trust interaction is tracked.

The koi case, precisely

  • Detect external management first: trust-hash entries in ~/.codex/config.toml, a koi marker, or hooks with a trusted_hash set.
  • Default to print for a managed file: emit the exact hook entry plus the trust step (/hooks in Codex). Mark the row ⚠ needs a manual step, not a check.
  • Edit only as a whole transaction: merge the entry and refresh the trust ledger so the hook stays trusted, and record externalOwner: "koi" so removal also cleans the trust entry. Never bypass trust.

Flags and removal

  • --dry-run prints the exact diff and writes nothing (conda / pipx):
signalbox init --agent codex --dry-run--- ~/.codex/hooks.json +++ ~/.codex/hooks.json (proposed) "PermissionRequest": [ ... koi ... , { "hooks": [{ "command": "signalbox hook codex" }] } ], + "Stop": [{ "hooks": [{ "command": "signalbox hook codex" }] }], + "SessionEnd": [{ "hooks": [{ "command": "signalbox hook codex" }] }] no changes written (--dry-run)
  • --yes never silently edits a user file. It applies signalbox-owned steps and prints for user files. To write user files non-interactively, pass --write-user-config as well (the rustup -y vs --no-modify-path gate).
  • signalbox init --reverse removes only signalbox's blocks and keys (by marker + manifest), restores backups, and revokes any trust entry it added - a first-class undo, so we never leave an orphaned edit the way gh does.

What stays the same

  • Files signalbox owns (the plugin symlinks) are still written directly, no prompt.
  • Print is still the default for anything shared or foreign - it is endorsed practice (Homebrew, starship, Claude Code's read-only /hooks). The change is that a printed step is honestly marked, and editing is now available behind consent for files we fully own the format of.
